# 랜덤 닉네임 생성 설계 고민

## 배경

신규 사용자에게 기본 닉네임을 자동으로 부여한다.
닉네임은 `수식어 + 동물` 조합을 사용한다.

예시:

```text
번듯한사자
조용한토끼
책읽는펭귄
```

현재 후보 수는 다음과 같다.

```text
수식어 71개 × 동물 75개 = 5,325개
```

초기에는 `랜덤 생성 + users.nickname UNIQUE + 최대 N회 재시도` 방식으로 접근했다.
하지만 후보 풀이 작고, 숫자 suffix를 붙이고 싶지 않기 때문에 중복과 고갈 정책을 함께 검토했다.

## 문제 정의

닉네임 생성에서 고려해야 할 문제는 다음과 같다.

- 같은 닉네임이 동시에 두 사용자에게 저장되면 안 된다.
- 운이 나빠서 재시도에 모두 실패했다는 이유로 가입이 막히면 이상하다. 하지만 지금의 조합으로는 충분히 유저를 감당이 가능하다.
- 해당 부분은 나중에 동물의 색을 결정하는 단어를 추가하거나 형용사와 동물의 수를 더 추가해서 보완이 가능하다
- 후보 풀이 5,325개로 크지 않다.
- 사용자가 닉네임을 변경하면 기존 닉네임은 다시 사용 가능해질 수 있다.
- 가입 요청이 쿠폰 이벤트처럼 짧은 시간에 대량으로 몰리는 도메인은 아니다.
- 필요하면 수식어, 동물, 색상, 분위기 단어를 추가해서 후보군을 쉽게 늘릴 수 있다.

## 확률 계산

전체 후보가 5,325개이고 이미 사용 중인 닉네임이 `U`개라면, 신규 사용자 1명이 랜덤으로 한 번 뽑았을 때 중복될 확률은 다음과 같다.

```text
중복 확률 = U / 5,325
```

예시:

| 사용 중인 닉네임 수 | 1회 생성 중복 확률 |
| ---: | ---: |
| 100개 | 약 1.88% |
| 300개 | 약 5.63% |
| 500개 | 약 9.39% |
| 1,000개 | 약 18.78% |

최대 10번 재시도한다면, 이미 300개가 사용 중일 때 10번 모두 중복될 확률은 다음과 같다.

```text
(300 / 5,325)^10 ≈ 0.000000000000322
```

초기 규모에서는 중복될 확률이 낮긴 하다.
다만 풀이 거의 찬 상황에서는 이야기가 달라진다.

남은 닉네임이 1개뿐이라면, 전체 후보를 랜덤 순서로 순회하는 방식에서도 성공 위치는 랜덤이다.

```text
최선: 1회 시도
평균: (5,325 + 1) / 2 = 2,663회 시도
최악: 5,325회 시도
```

이때 병목은 문자열 조합 비용이 아니라 DB insert 실패와 unique 충돌 처리 비용이다.

## 선택지

### 1. 랜덤 생성 + UNIQUE + 제한 재시도

가입 시점에 닉네임을 랜덤으로 생성하고 `users.nickname` unique 제약으로 중복을 막는다.
충돌하면 다시 생성한다.

장점:

- 구현이 단순하다.
- 별도 테이블이나 seed migration이 필요 없다.
- 초기 사용자 수가 적으면 충분히 실용적이다.

단점:

- 이론상 제한 횟수 안에 모두 충돌하면 가입 실패가 발생한다.
- 풀이 찰수록 충돌 확률이 높아진다.
- 실패 원인이 "후보 고갈"인지 "운이 나쁜 재시도"인지 명확하지 않다.

### 2. 전체 후보 캐싱 + 랜덤 순회

앱 시작 후 전체 조합 5,325개를 한 번 만들고 메모리에 보관한다.
신규 가입 시 후보 리스트를 랜덤 순서로 순회하면서 insert를 시도한다.

장점:

- 10번 운 나쁘게 실패해서 가입이 막히는 문제를 줄일 수 있다.
- DB 테이블을 추가하지 않아도 된다.
- 후보 수가 5,325개면 메모리 부담은 작다.

단점:

- 후보 풀이 거의 찬 상태에서는 평균 DB 시도 횟수가 커질 수 있다.
- 서버 인스턴스마다 같은 후보 리스트를 들고 있으므로 최종 중복 방어는 여전히 DB unique 제약이 담당해야 한다.
- race condition 자체를 없애는 방식은 아니다.

### 3. nickname_pool 테이블

완성된 닉네임 조합 5,325개를 미리 DB에 저장해둔다.
가입 시 `assigned_user_id IS NULL`인 닉네임 하나를 락으로 잡아 할당한다.

예시 테이블:

```sql
CREATE TABLE nickname_pool (
    id BIGSERIAL PRIMARY KEY,
    nickname VARCHAR(15) NOT NULL UNIQUE,
    assigned_user_id BIGINT,
    assigned_at TIMESTAMP
);
```

예시 할당 쿼리:

```sql
SELECT id, nickname
FROM nickname_pool
WHERE assigned_user_id IS NULL
ORDER BY random()
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

장점:

- 남은 후보 중에서만 가져오므로 중복 재시도 비용이 작다.
- 남은 닉네임 수를 운영 지표로 볼 수 있다.
- 동시성 제어를 DB row lock으로 명확히 표현할 수 있다.
- 후보가 거의 찬 상황에서도 안정적이다.

단점:

- 테이블, seed migration, 할당 로직, 반납 정책이 추가된다.
- 닉네임 변경 또는 탈퇴 시 기존 닉네임을 반환할지 정책이 필요하다.
- 단어를 추가하면 조합 seed를 다시 관리해야 한다.
- 현재 도메인에 비해 운영 복잡도가 커질 수 있다.

`nickname_pool`은 DB 테이블이므로 후보 전체를 반드시 애플리케이션 메모리에 들고 있을 필요는 없다.
다만 "미리 만든 후보 전체를 별도 리소스로 관리한다"는 비용은 생긴다.
반대로 코드에서 전체 후보를 캐싱하는 방식은 실제로 애플리케이션 메모리를 사용한다.

### 4. 후보군 확장

후보 수가 작다는 문제는 조합 축을 하나 더 추가해서 완화할 수 있다.

예시:

```text
푸른조용한토끼
하얀책읽는사자
노란산책하는펭귄
```

색상 또는 분위기 단어 30개를 추가하면 후보 수는 다음과 같이 늘어난다.

```text
30 × 71 × 75 = 159,750개
```

후보 수가 커지면 `랜덤 생성 + UNIQUE + 재시도` 방식의 실용성이 크게 올라간다.

## race condition

애플리케이션에서 랜덤 닉네임을 생성하는 것만으로는 동시성 문제가 해결되지 않는다.

예시:

```text
요청 A: 조용한토끼 생성
요청 B: 조용한토끼 생성
```

두 요청이 동시에 같은 닉네임을 만들 수 있다.
따라서 최종 방어선은 반드시 DB unique 제약이어야 한다.

```text
요청 A: insert 성공
요청 B: unique 충돌
요청 B: 다음 후보로 재시도
```

`nickname_pool` 방식도 동시성 제어가 필요하다.
이 경우에는 `FOR UPDATE SKIP LOCKED` 같은 row lock으로 같은 후보를 동시에 잡지 않게 만들 수 있다.

## 쿠폰 코드와의 비교

랜덤 닉네임 문제는 쿠폰 코드 발급 문제와 겉으로는 비슷하다.
둘 다 "유한한 후보 풀에서 중복 없이 하나를 발급한다"는 형태를 가진다.

하지만 도메인 특성이 다르다.

### 쿠폰 코드의 특징

쿠폰 코드는 보통 다음 특징을 가진다.

- 유한한 재고다.
- 중복 발급이 비용 사고로 이어질 수 있다.
- 이벤트나 푸시 발송 시 짧은 시간에 대량 발급될 수 있다.
- 발급, 예약, 사용, 만료, 취소 같은 상태 전이가 필요하다.
- 누가 언제 발급받고 사용했는지 감사 로그가 중요하다.
- 사용자가 코드를 예측하면 안 되므로 랜덤성이 중요하다.
- 남은 수량을 운영 지표로 관리해야 한다.

쿠폰 상태는 다음처럼 모델링될 수 있다.

```text
AVAILABLE → RESERVED → ISSUED → REDEEMED
                         → EXPIRED
                         → CANCELLED
```

이런 특성 때문에 쿠폰은 pre-generated pool 방식이 잘 맞는다.
미리 쿠폰 코드를 만들어두고, 발급 시 미사용 코드 하나를 원자적으로 할당하는 구조가 자연스럽다.

### 닉네임의 특징

닉네임은 쿠폰보다 느슨하다.

- 비용성 재고가 아니다.
- 사용자가 변경하면 기존 닉네임이 다시 사용 가능해질 수 있다.
- 대량 이벤트처럼 빠르게 소진되는 성격이 약하다.
- 후보군을 쉽게 확장할 수 있다.
- 남은 닉네임 수를 엄격한 운영 지표로 볼 필요가 낮다.
- 닉네임은 표시 이름이고, 실제 식별자는 user id가 담당한다.

따라서 쿠폰에서 좋은 설계가 닉네임에도 항상 정답은 아니다.

## 현재 판단

현재 도메인에서는 `nickname_pool`이 정답이라고 보기 어렵다.

이유:

- 가입자가 매우 빠르게 대량 유입되는 상황이 아니다.
- 후보 조합은 쉽게 늘릴 수 있다.
- 닉네임은 변경될 수 있고, 기존 닉네임은 다시 사용 가능해질 수 있다.
- 쿠폰처럼 소진 자체가 비용 사고로 이어지지 않는다.
- `nickname_pool`을 만들면 테이블, seed, 할당, 반납, 운영 정책이 추가된다.

따라서 우선순위는 다음이 적절하다.

```text
1. 단어 목록은 resources/nickname-words.yml에서 관리한다.
2. users.nickname UNIQUE 제약은 유지한다.
3. 랜덤 생성 후 unique 충돌 시 재시도한다.
4. 재시도 실패가 우려되면 후보군을 확장한다.
5. 정말 후보 고갈이나 운영 가시성이 필요해질 때 nickname_pool을 검토한다.
```

후보군 확장은 `nickname_pool`보다 단순하고 도메인에도 잘 맞는다.

예시:

```text
2단 조합: 조용한토끼
3단 조합: 푸른조용한토끼
```

처음부터 3단 조합을 전부 쓰거나, 충돌이 반복될 때만 3단 조합으로 fallback하는 방식도 가능하다.

## 현재 구현

현재 구현은 `nickname_pool`을 사용하지 않는다.
단어 목록은 `app/src/main/resources/nickname-words.yml`에서 관리하고, 애플리케이션 시작 시 loader가 읽어 검증한다.

OAuth 가입/로그인 흐름은 다음과 같다.

```text
1. nickname-words.yml에서 수식어와 동물 목록을 읽는다.
2. OAuth 요청마다 닉네임 후보 1개를 랜덤 생성한다.
3. users에 insert ... on conflict (provider, provider_id) do update를 실행한다.
4. provider + provider_id 충돌은 기존 사용자 로그인으로 자연스럽게 처리한다.
5. nickname unique 충돌만 잡아서 닉네임을 다시 생성하고 재시도한다.
6. 정해진 재시도 횟수를 넘기면 NICKNAME_GENERATION_FAILED를 반환한다.
```

중요한 점은 `ON CONFLICT DO NOTHING`으로 모든 unique 충돌을 삼키지 않는다는 것이다.
`provider + provider_id` 충돌은 upsert가 처리하고, `nickname` 충돌은 예외로 올려 재시도 대상인지 명확히 구분한다.

## 블로그 글감으로 정리하기

블로그로 작성한다면 핵심 메시지는 다음이 좋다.

```text
같은 랜덤 중복 문제라도 도메인 특성에 따라 정답이 달라진다.
쿠폰은 pre-generated pool이 자연스럽지만, 닉네임은 unique 제약 + 재시도 + 후보군 확장이 더 단순할 수 있다.
```

추천 글 흐름:

1. `형용사 + 동물` 닉네임 생성 요구사항 소개
2. 후보 수 `71 × 75 = 5,325개` 계산
3. 기존 사용자 수에 따른 중복 확률 계산
4. `UNIQUE + retry` 방식의 장단점
5. 후보 풀이 찰수록 충돌이 늘어나는 문제 소개
6. 쿠폰 코드 도메인에서는 왜 pre-generated pool이 잘 맞는지 설명
7. 닉네임 도메인에서는 왜 `nickname_pool`이 과할 수 있는지 설명
8. 최종 선택과 나중에 바꿀 조건 정리

실험을 곁들이면 다음 방식을 비교할 수 있다.

```text
1. 랜덤 생성 + 10회 재시도
2. 전체 후보 shuffle 후 순회
3. nickname_pool에서 미사용 row 할당
4. 후보군을 3단 조합으로 확장한 뒤 랜덤 생성 + 재시도
```

측정 지표:

```text
사용률 0%, 50%, 90%, 99%, 99.9%에서
평균 시도 횟수
최악 시도 횟수
평균 응답 시간
동시 요청 시 중복 저장 여부
가입 실패 여부
```
